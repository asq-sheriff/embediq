## What Is This?

EmbedIQ, by Praglogic, is an intelligent configuration wizard that interviews you about your project, team, and goals — then generates a complete, production-ready AI coding agent harness tailored to your specific needs.

It adapts to who you are. A solo developer building a SaaS app gets a lean, productivity-focused setup. A healthcare enterprise team gets HIPAA compliance hooks, PHI detection, audit trails, and security-hardened permissions. A business analyst gets an "agent as intelligent coworker" setup optimized for research, analysis, and documentation — not code.

## Getting Started — Step by Step

### Step 1: Install EmbedIQ

Clone the repository and install dependencies:

```bash
git clone <repo-url>
cd embediq
npm install
```

This takes about 30 seconds and downloads ~50 MB of Node.js packages.

### Step 2: Launch the Wizard

Choose your preferred interface:

**Option A — Web UI** (recommended for all users):

```bash
npm run start:web
```

You should see:

```
  ┌─────────────────────────────────────────┐
  │                                         │
  │   EmbedIQ by Praglogic                  │
  │   Claude Code Setup Wizard              │
  │                                         │
  │   http://localhost:3000                  │
  │                                         │
  └─────────────────────────────────────────┘
```

Open `http://localhost:3000` in your browser.

**Option B — CLI** (for terminal-native developers):

```bash
npm start
```

The CLI walks you through the same four phases interactively in your terminal.

### Step 3: Complete the Wizard

EmbedIQ guides you through four phases (detailed in the next section). The entire process takes 5–15 minutes depending on the complexity of your setup. At the end, you specify a target project directory and EmbedIQ writes 15–40 configuration files.

### Step 4: Install Claude Code (if not already installed)

```bash
npm install -g @anthropic-ai/claude-code
```

### Step 5: Activate Your Configuration

Navigate to the project where EmbedIQ generated the files and start Claude Code:

```bash
cd /path/to/your/project
claude
```

Claude Code automatically detects and loads CLAUDE.md, all rules in `.claude/rules/`, hook scripts, and agent definitions. No additional setup is needed.

### Step 6: Configure MCP Servers (optional)

If the wizard generated an `.mcp.json.template` file:

```bash
cp .mcp.json.template .mcp.json
```

Edit `.mcp.json` and replace placeholder values (`${GITHUB_TOKEN}`, `${DATABASE_URL}`, etc.) with your actual credentials. This file is gitignored — your secrets stay local.

### Step 7: Commit Shared Configuration to Git

Add the files that should be shared with your team:

```bash
git add CLAUDE.md .claude/settings.json .claude/rules/ .claude/commands/ \
       .claude/agents/ .claude/skills/ .claudeignore .claude/.claude_ignore \
       .mcp.json.template
git commit -m "Add Claude Code configuration generated by EmbedIQ"
```

**Do not commit** (these are local/sensitive and should be gitignored):
- `.claude/settings.local.json` — personal permission rules
- `.mcp.json` — contains API keys
- `.claude/logs/` — audit logs

## The Four Phases

EmbedIQ guides you through four phases. Each builds on the previous one.

### Phase 1 — Discovery

The wizard asks 25–40 questions (from a bank of 71) organized across seven dimensions:

| Dimension | What It Covers |
|---|---|
| Strategic Intent | Your role, industry, project purpose, goals, criticality |
| Problem Definition | Pain points, quality gates, TDD preference, audience |
| Operational Reality | Team size, branching strategy, parallel work, code review |
| Technology Requirements | Languages, frameworks, IDEs, build tools, CI/CD, databases, MCP servers |
| Regulatory Compliance | PHI/PII handling, DLP controls, audit trails, permissions, data classification |
| Financial Constraints | API budget, model routing strategy, local model preference |
| Innovation & Future-Proofing | Plugins, doc tracking, association maps, memory, agents, commands |

**Adaptive branching** means you only see questions relevant to you:

- Select "Healthcare" as your industry → HIPAA and PHI questions appear
- Select "Solo developer" → team coordination questions are skipped
- Select "Business Analyst" as your role → deeply technical questions (build tools, linting, TDD) are hidden entirely
- Say you handle sensitive data → cascading compliance questions unlock (DLP patterns, audit storage, context sanitization, ZDR, network egress)
- Say budget is minimal → local model and cost optimization questions surface

### Phase 2 — Playback

The wizard summarizes what it understood:

- Your profile (role, industry, team, stack)
- Your DevOps environment (IDE, build, test, CI/CD, monitoring)
- Compliance frameworks that apply
- Security concerns detected
- **Interpreted priorities** — ranked with confidence scores (e.g., "Security & Compliance: 95%", "Cost Optimization: 72%")

Priorities are derived from your answers using a tag-weight analysis system. Each question carries semantic tags (e.g., `security`, `compliance`, `phi`). Your answers accumulate weight across eight priority categories:

| Category | What Triggers It |
|---|---|
| Security & Compliance | Security concerns, PHI/PII handling, DLP, audit trails |
| Cost Optimization | Budget constraints, local model preference, model routing |
| Code Quality | Testing emphasis, TDD, linting, quality gates |
| Developer Productivity | Command/agent interest, automation, devtools |
| Team Coordination | Team size, parallel work, code review, branching |
| CI/CD & Automation | CI/CD platform, containers, deployment |
| Monitoring & Observability | Monitoring tools, logging, audit logging |
| Documentation & Knowledge | Memory, association maps, document lifecycle |

Confidence is calculated as: `total weight from matching tags / (number of matching tags x 5)`. Boolean "yes" answers contribute 3 points, scale answers contribute their value (1-5), multi-choice answers contribute `min(count x 1.5, 5)`, and non-empty text contributes 2. Categories below 10% confidence are filtered out.

### Phase 3 — Edit, Correct, Approve

Before anything is generated, you can:

- **Correct** any field (domain, industry, team size, languages, budget, CI/CD)
- **Adjust priorities** — promote a priority to the top if the wizard misjudged
- **Add missing items** — add a compliance framework, language, or security concern the wizard didn't capture

This loop repeats until you approve.

### Phase 4 — Generate

You specify a target project directory. The wizard generates 15–40 configuration files directly into that directory, creating the full `.claude/` structure. Every file is shown with a success/failure indicator and description.

## Agents

Agents are isolated Claude Code sub-sessions with restricted tools and specific model assignments. They run independently from your main conversation, which means they can't accidentally modify your working files (unless explicitly granted write access).

To receive agents, you must answer **"yes"** to the agent enablement question during the Innovation dimension. Then, specific agents are generated based on your profile:

| Agent | What It Does | Model | Tools | Isolation |
|---|---|---|---|---|
| **security-reviewer** | Scans code for OWASP vulnerabilities, hardcoded secrets, PHI/PII exposure | Opus | Read, Grep, Glob, Bash (read-only) | None |
| **compliance-checker** | Validates code against your compliance frameworks (HIPAA, SOC2, PCI-DSS, GDPR) | Opus | Read, Grep, Glob (read-only) | None |
| **code-reviewer** | Reviews code changes against project conventions and quality standards | Sonnet | Read, Grep, Glob (read-only) | None |
| **test-writer** | Generates tests for uncovered code, following existing test patterns | Sonnet | Read, Grep, Glob, Write, Edit | Worktree |

The test-writer uses **worktree isolation** — it creates and modifies files in a temporary git worktree, so failed experiments don't affect your working branch.

To invoke an agent in Claude Code:

```
/agent security-reviewer
/agent test-writer
```

## Skills

Skills are multi-step workflows that run within your main Claude Code conversation (no isolation). They combine several operations into a single invocable procedure.

| Skill | What It Does | Model |
|---|---|---|
| **sync-memory** | Reads `.claude/memory/` files, compares with current codebase state, updates stale entries, reports changes | Haiku |

To invoke a skill in Claude Code:

```
/skill sync-memory
/skill impact-analysis
```

## Security & Compliance Features

### How Hooks Work

Hooks are Python scripts that Claude Code runs automatically when you use certain tools. They execute **before** (PreToolUse) or **after** (PostToolUse) a tool action and control whether the action proceeds.

The execution flow:

```
You ask Claude to edit a file
        │
        ▼
Claude calls the Edit tool
        │
        ▼
Claude Code runs PreToolUse hooks (e.g., dlp-scanner.py)
        │
        ├── Hook reads tool input as JSON from stdin
        ├── Hook scans for patterns (secrets, PHI, dangerous commands)
        ├── Hook prints findings to stderr (shown to you)
        │
        ▼
Hook returns an exit code
        │
        ├── Exit 0 (PASS)  → Tool executes normally
        ├── Exit 1 (WARN)  → Warning shown, tool still executes
        └── Exit 2 (BLOCK) → Tool execution prevented, you see the reason
```

Hooks are generated as Python scripts in `.claude/hooks/` and registered in `.claude/settings.json`. They require **Python 3.8+** at runtime.

### Data Loss Prevention (DLP)

When enabled, the DLP scanner hook runs as a `PreToolUse` hook on every Edit, Write, and Bash operation. It scans for:

| Pattern | Severity | Default |
|---|---|---|
| API keys and tokens | CRITICAL | Always |
| AWS access keys | CRITICAL | Always |
| Private keys/certificates | CRITICAL | Always |
| Social Security Numbers | CRITICAL | When PHI or PII enabled |
| Credit card numbers | CRITICAL | When PHI or PII enabled |
| Medical Record Numbers | CRITICAL | When PHI enabled |
| Patient data fields | HIGH | When PHI enabled |
| Custom patterns | Configurable | When user provides regex |

The scanner uses exit code 2 (BLOCK) for critical matches — Claude Code cannot proceed until the sensitive data is removed.

### Permission Tiers

Four security levels, selected during the wizard:

| Level | Allow | Deny | Best For |
|---|---|---|---|
| Permissive | Most git, npm, docker, test commands | Destructive ops only | Solo dev, low risk |
| Balanced | Safe git reads, npm run/test, npx, ls | Destructive + credential files | Most teams |
| Strict | git status, git diff, git log, ls only | Destructive + network + credentials | Regulated environments |
| Lockdown | git status, ls only | Everything else including curl, wget, ssh | Maximum security |

### Audit Logging

When enabled, every Claude Code tool action is logged to `.claude/logs/audit-YYYY-MM-DD.jsonl` with:

- Timestamp (UTC)
- User identity
- Tool name
- Summarized input (sensitive content is replaced with type descriptors)
- Session ID

### Context Sanitization

Sensitive directories listed in `.claudeignore` are excluded from Claude's context entirely. Combined with deny rules in `settings.local.json`, this creates a multi-layer protection architecture:

1. **Layer 1**: `.claudeignore` prevents files from entering context
2. **Layer 2**: `PreToolUse` hooks scan for sensitive patterns in real-time
3. **Layer 3**: `settings.local.json` deny rules block writes to protected paths

## Association Map

When enabled, EmbedIQ generates `association_map.yaml` — a bidirectional map that links related files across four categories:

| Category | What It Contains |
|---|---|
| **Code** | Source file patterns, adapted to your languages (`src/**/*.ts`, `app/**/*.py`, etc.) |
| **Tests** | Test file patterns, adapted to your frameworks (`**/*.test.ts` for Jest, `tests/**/*.py` for pytest, etc.) |
| **Docs** | Documentation patterns (`docs/**/*.md`, `CLAUDE.md`, `README.md`, rule files) |
| **Infrastructure** | Build configs, CI/CD files, Dockerfiles, adapted to your build tools |

The association map is used by the **impact-analysis** skill to determine which tests, docs, and infrastructure files might need updating when code changes. It ships with starter patterns and an example entry — customize it by adding your own associations:

```yaml
associations:
  - description: User authentication service
    code: ['src/auth/**']
    tests: ['tests/auth/**']
    docs: ['docs/auth.md']
    infrastructure: ['.github/workflows/auth-tests.yml']
```

## Document State Tracking

When enabled, EmbedIQ generates `docs/document_state.yaml` — a lifecycle registry that categorizes each documentation file by its current state:

| State | Meaning | When to Use |
|---|---|---|
| **CURRENT** | Reflects deployed reality | Must be kept in sync with code changes |
| **FUTURE** | Planned or roadmap content | Not yet implemented; don't treat as truth |
| **REFERENCE** | Timeless reference material | Rarely needs updates (style guides, glossaries) |
| **ARCHIVED** | Superseded or obsolete | Kept for historical context only |

Tools and skills use this registry to determine which docs need updates when code changes — CURRENT docs are candidates for update, FUTURE and ARCHIVED docs are skipped.

## Model Routing & Cost Optimization

When model routing is enabled, the wizard generates slash commands that route tasks to the appropriate model tier:

| Command | Model | Cost | Use For |
|---|---|---|---|
| `/quick` | Haiku | ~10x cheaper than Opus | Lookups, explanations, status checks |
| `/code` | Sonnet | ~3x cheaper than Opus | Writing code, refactoring, debugging |
| `/think` | Opus | Full price | Architecture, security audits, complex analysis |

Additional cost controls:

- **OpusPlan**: Opus for planning, Sonnet for execution (~68% savings)
- **SonnetPlan**: Sonnet for planning, Haiku for execution (maximum savings)
- **Subagent model override**: `CLAUDE_CODE_SUBAGENT_MODEL=haiku` (~80% cheaper subagents)
- **Thinking token limits**: `MAX_THINKING_TOKENS=10000` (~70% reduction in hidden costs)
- **Local model delegation**: Ollama MCP server for linting, formatting, simple queries

## Data Privacy & Security

EmbedIQ is designed with a **zero-persistence, zero-telemetry architecture**. This section documents exactly what happens to your data at every stage.

### What Data Does EmbedIQ Collect?

During the wizard, you provide answers about your project, team, industry, compliance requirements, and technology stack. In regulated domains (healthcare, finance), some of these answers constitute **sensitive metadata**:

| Data Category | Examples | Sensitivity |
|---|---|---|
| Role & industry | "Healthcare", "Finance" | Low — general context |
| Compliance frameworks | "HIPAA", "PCI-DSS" | Medium — reveals regulatory obligations |
| PHI/PII handling details | DLP patterns, data classification levels, audit storage preferences | **High — sensitive metadata** indicating what types of protected data your organization handles |
| Custom DLP regex patterns | User-supplied patterns for detecting PHI/PII | **High — reveals the shape of protected data your systems process** |
| Security posture | Permission tiers, egress controls, protected file paths | **High — reveals security architecture** |
| Technology stack | Languages, frameworks, CI/CD, databases | Low-Medium — could reveal attack surface |

### Where Does This Data Go?

**Nowhere.** EmbedIQ operates entirely locally:

| Stage | Where Data Lives | Duration |
|---|---|---|
| CLI wizard | In-process memory (Node.js heap) | Until process exits |
| Web wizard (browser) | Browser JavaScript memory | Until tab is closed or refreshed |
| Web wizard (server) | Request-scoped — lives only for the duration of a single HTTP request | Milliseconds |
| Generated files | Written to local disk at your specified target directory | Permanent (you control) |

There is **no**:
- Database (no SQLite, no PostgreSQL, no file-based store)
- Server-side session storage
- Cookies or local storage
- Telemetry, analytics, or usage tracking
- Network calls to external services (EmbedIQ never phones home)
- Logging of user answers (the server logs only startup messages)

### What Does NOT Touch an LLM?

This is a critical distinction. EmbedIQ itself **never sends your answers to any LLM**. The wizard is deterministic — it uses conditional branching logic and template-based generation, not AI inference. Your answers about PHI handling, DLP patterns, compliance frameworks, and security posture are processed entirely by local code.

The generated configuration files (CLAUDE.md, rules, hooks, etc.) are later consumed by Claude Code, which does use an LLM. However:

- **CLAUDE.md and rules** contain instructions and conventions — not your data. They tell Claude *how to behave*, not what your PHI looks like.
- **Hook scripts** run locally as Python processes. They scan for patterns but don't send data to any LLM.
- **DLP regex patterns** are embedded in local Python scripts, not sent to Anthropic.
- **Permission deny rules** are enforced locally by Claude Code before any tool action reaches the LLM.

### Data Flow Diagram

```
Your Answers (in-memory only)
        │
        ▼
```

### Regulated Domain Considerations

For organizations subject to HIPAA, PCI-DSS, SOC2, GDPR, or similar frameworks:

**During the wizard:**
- Answers about PHI/PII handling, DLP patterns, and compliance posture are **sensitive metadata**. They reveal what types of protected data your organization processes and how you protect it.
- EmbedIQ does not persist, log, or transmit these answers. They exist only in volatile memory for the duration of the wizard session.
- For maximum assurance, run EmbedIQ on a local machine (not a shared server) when answering compliance-related questions.

**After generation:**
- Review generated files before committing to git. Files like `.claude/hooks/dlp-scanner.py` contain your custom DLP regex patterns — these reveal the shape of your protected data.
- The `.mcp.json.template` file may contain server names and service endpoints. Never commit the filled-in `.mcp.json`.
- Audit trail configuration (in `audit-logger.py`) writes to local files only. No audit data is sent externally.

**What to keep out of git** (already handled by generated `.gitignore` patterns):
- `.claude/settings.local.json` — contains permission rules that reveal security posture
- `.mcp.json` — contains API keys and service URLs
- `.claude/logs/` — contains audit trail data

### Web Server Authentication

When running the web UI, EmbedIQ supports optional basic authentication to prevent unauthorized access. This is important when:

- Running on a shared network or server
- The wizard will collect compliance/security metadata
- Organizational policy requires access controls on configuration tools

Enable authentication by setting environment variables:

```bash
EMBEDIQ_AUTH_USER=admin EMBEDIQ_AUTH_PASS=your-secure-password npm run start:web
```

When set, the web server requires HTTP Basic Authentication for all routes. When not set, the server runs without authentication (suitable for local-only use).

For team deployments, consider running behind a reverse proxy (nginx, Caddy) with your organization's SSO/SAML provider for enterprise-grade authentication.

## Troubleshooting

### "DLP SCAN RESULTS" blocking my edit

The DLP scanner detected a pattern matching sensitive data (SSN, credit card, API key, etc.). Remove the sensitive data from your edit. If it's a false positive in test data, add the test directory to the hook's whitelist in `dlp-scanner.py`.

### Questions seem too technical / not technical enough

The wizard adapts based on your role selection (first question). If you selected the wrong role, restart the wizard and choose the role that best matches your work.

### Generated files conflict with existing config

The wizard writes directly to your target directory. If you already have a `CLAUDE.md` or `.claude/` directory, the wizard will overwrite those files. Back up existing configuration before running.

### Web server won't start

Check that port 3000 (or your configured port) is not in use. Try `PORT=3001 npm run start:web`.

### Build errors

Run `npm install` to ensure all dependencies are installed, then `npm run build` to check for TypeScript compilation errors.
